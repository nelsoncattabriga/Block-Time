## What Was Done

### Task 1 — BackupsView alert logic (HIGH)

Three independent structs each had `.alert(...)` titles computed by scanning `resultMessage` for substrings like `"successfully"`, `"Restored"`, `"Summary"`. This was fragile — a message change could silently flip a success alert to an error title.

Added `@State private var isSuccess = false` to each struct. Each success/failure branch now sets `isSuccess` before the show-alert flag. Alert titles use `isSuccess ? "Success" : "Error"`.

- `BackupsView.createManualBackup()` — 2 assignments
- `BackupDetailSheet.executeRestore()` — 2 assignments; dismiss check also updated from string scan to `isSuccess`
- `ManageBackupsView.executeExternalRestore()` — 2 assignments

### Task 2 — ImportMappingView (MEDIUM + LOW)

Removed 4 debug `print()` calls that fired on every view render (3 in `init`, 1 in `body`) plus 1 in the static `createInitialMappings` method. These caused unnecessary console spam on every navigation to this view.

Replaced 8 deprecated `.navigationBarLeading`/`.navigationBarTrailing` toolbar placements with `.topBarLeading`/`.topBarTrailing` across `ImportMappingView`, `ColumnPickerView`, `RegistrationTypeMappingView`, and `ManageCustomTypesView`.

### Task 3 — AutomaticBackupService and ExportLogbookView (MEDIUM + LOW)

**DateFormatter hoisting:** Both files were allocating a `DateFormatter()` inside the sort closure comparator — once per comparison call, which for large logbooks could be thousands of allocations per sort. Moved to a single `flightSortFormatter` local created once before the `.sorted` call.

**loggerDir removal:** `getBackupDirectory()` had a vestigial `loggerDir` variable (a commented-out `.appendingPathComponent("Block-Time")` that was never applied). Removed the intermediate variable; `backupDir` now comes directly from `baseURL.appendingPathComponent("Backups")`.

**Task.sleep removal:** `ExportLogbookView.performExport()` had a 300ms cosmetic sleep at the top of the `do` block. After removing it, the remaining code inside the `Task` closure had no other `try` expressions (the CSV write is the only throw but still uses `try`), so the `do` wrapper was kept to handle the `try csvString.write(...)` call.
