<objective>
Fix the stale "Custom Count" column appearing in FrozenColumnSpreadsheetView when "Use Custom Fields" is ON but no CustomCounterDefinition entries exist (and other edge cases that show the legacy column after migration).

Purpose: The legacy single `customCount` column should never appear once the multi-counter system is active. The visible counter columns must reflect exactly what the user has configured via `CustomCounterService.shared.definitions`, gated by the `useCustomCount` toggle.

Output: A corrected FrozenColumnSpreadsheetView where header, totals row, horizontal scroll width, and right-cell value rendering all use a single, consistent rule for counter columns — with the legacy `customCount` column removed as a visible artefact.
</objective>

<bug_description>
Reported behaviour: "Use Custom Fields" toggle is ON, but no field definitions exist yet → spreadsheet still shows a "Custom Count" column (the legacy single-counter column). This is stale and confusing.

Root cause: `FrozenColumnSpreadsheetView.swift` has fallback branches in `makeRightHeader()` (lines 441-450, 488-497) and `RightCell.configure()` (lines 785-803) that render the legacy `customCount` column when `counterDefinitions.isEmpty`. The `useLegacyColumn` flag only protects against the post-migration case, not the "ON + no definitions" case. Also `useCustomCount` UserDefaults flag is not consulted at all in this view.
</bug_description>

<expected_logic>
Single rule for counter columns:

```
useCustomFields = UserDefaults.standard.bool(forKey: "useCustomCount")
definitions     = CustomCounterService.shared.definitions
visibleCounterDefinitions = useCustomFields ? definitions : []
counterCount = visibleCounterDefinitions.count   // 0 is valid; do NOT force max(_, 1)
```

The legacy `customCount` column is NEVER rendered (the task explicitly states once `legacyCounterMigratedToColumn1` is set the legacy column must never appear; and any new install will have the migration flag set on first launch of multi-counter code). Remove the `useLegacyColumn` branch entirely from header + totals + RightCell.

Note: The migration flag itself is no longer needed for gating in THIS view — `useCustomCount` + definitions array length covers all cases. Do not touch the migration flag, UserDefaults keys, CustomCounterService API, or `counter1`–`counter10` attribute names per constraints.
</expected_logic>

<width_fix>
`Col.rightWidth(counterCount:)` currently uses `counter * CGFloat(max(counterCount, 1))` which reserves a phantom column when `counterCount == 0`. Change to `counter * CGFloat(counterCount)` so zero counters means zero counter width.

`RightCell.rebuildSubviews(counterCount:)` already handles 0 correctly (empty `Array(repeating:count:0)`), but verify it tracks `builtCounterCount == 0` distinct from initial `-1` (it does — keep as-is).
</width_fix>
